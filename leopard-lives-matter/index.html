Obtaining the Data¶

First order of business is to obtain the killmails seen in the screenshot above. We cannot use ESI to query for killmails of other users but we can use the zKillboard API to scrape whatever killmals are available. The /kills/characterID/{character_id}/ endpoint returns a list of recent kills from character with CCP ID {character_id}. The list consists of entries in the following format:

{
    "killmail_id": 1234,
    "zkb": {
        "locationID": 1234,
        "hash": "foobarbaz",
        // other irrelevant fields
    }
}

locationID is the ID of the celestial closest to the spot where the kill ocurred. We need the combination of killmail_id and hash to fetch the actual killmails from ESI. Since we are only interested in kills that happen in Jita, we will obtain the list of location IDs in Jita and use it to filter the list from zKillboard before we turn to ESI to download the actual killmails. ESI provides the /universe/systems/{system_id}/ endpoint for querying system information. Let's see what's in Jita:

A lot of interesting information here. attackers is the list of attackers, including the type of ship they flew. The most interesting information though is contained in the victim field and that is the position field holding the 3d space coordinates of the location where the kill ocurred. We also have the ship type of the victim, something that the zKillboard data did not provide.

Let's write a function that downloads a killmail, checks if the victim was a Leopard in Jita and if so, returns the time and location of the kill. Also, we are only interested in solo kills with specific kind of thermal smartbomb as the poor Leopard has a thermal resistance hole.

Since we are going to be playing with a lot of ESI calls, it is a good idea to cache them locally so that we do not unnecessary hit the API server. The information we are interested in is static, so we also save and load the ESI cache by pickling it in order to reuse it in different sessions.

In [7]:
import time

ESI_CACHE = {}

try:
    import pickle
    with open('esi.cache', 'rb') as f:
        ESI_CACHE = pickle.load(f)
except FileNotFoundError:
    pass

def cached_esi_call(url):
    global ESI_CACHE
    if url not in ESI_CACHE:
        r = req.get(url)
        time.sleep(1.0)
        if r.status_code != 200:
            print("While fetching", url)
            print("got bad HTTP response from ESI:", r.status_code)
            raise ConnectionError()
        ESI_CACHE[url] = r.json()
    return ESI_CACHE[url]

def get_killmail_if_interesting(km_id, km_hash):
    km = cached_esi_call(f"https://esi.evetech.net/latest/killmails/{km_id}/{km_hash}/")
    attackers = km['attackers']
    # Should not happen but let's safeguard anyway
    if km['solar_system_id'] != LID_JITA:
        print("Location is not Jita - check filter list")
        return None
    # Only solo kills
    if len(attackers) > 1:
        print("Not a solo killmail")
        return None
    # Only large thermal smartbombs
    if attackers[0]['weapon_type_id'] != TID_YFLARGE:
        print("Didn't die to a large YF-12a smartbomb")
        return None
    # Only Leopards
    victim = km['victim']
    if victim['ship_type_id'] != TID_LEOPARD:
        print("Not a Leopard")
        return None
    # Extract useful information
    victim_pos = km['victim']['position']
    return {
        'id': km_id,
        'time': km['killmail_time'],
        'x': victim_pos['x'],
        'y': victim_pos['y'],
        'z': victim_pos['z']
    }

ESI may rate-limit us if we hit it too hard, which is why we insert a small 1 second delay after each request. We do not want to deal with rate limit errors, or with any other kind of errors, so we simply raise an exception when that happens. Before venturing forth, we need to talk about how to locate things in EVE.

The Geometry of EVE¶

In their desire to create as realistic a simulation as possible, CCP went as far as representing the position of each star in EVE in metres on a shared galactic Cartesian coordinate system. to keep the position of each object in the EVE universe with very high accuracy down to a single metre. The objects within the solar systems (planets, moon, stations, etc.) use the same coordinate system but have their positions relative to their star. This allows for really precise computations using the game location data.

Because of the huge distances involved, working with the standard floating-point type of Python will not yield good results in some cases due to errors introduced by rounding. There, when that makes sense, we will use the decimal package that provides more textbook style, arbitrary precision decimal arithmetic.

Although we know the exact 3d coordinates of the spot, EVE does not allow us to randomly warp anywhere in space. Instead, we can only warp to some of a predefined set of warp targets such as celestials, stations, gates, beacons, wrecks or bookmarked locations more than 150 km away from our current location. It is possible to warp only halfway through to the warp target by first depleting the ship's capacitor until there isn't enough charge left for the full warp, but that is difficult to calculate in advance and capacitor levels are quantised. Another option is to create a bookmark in mid-warp and then warp back to it, which is how most safespots are created. As we will see later on, bookmarks are also quantised, which complicates things.

Because of that peculiarity of the warp mechanic, there is a limited number of "official" warp trajectories that pass through $P_0$. Given the sheer number of killmails, we can safely assume that we are dealing with one such official trajectory, most likely one connecting a gate to the Jita 4-4 market. But let's not get ahead of ourselves and examine the situation thoroughly.

We are going to brute force all pairs of space objects. For each pair, we will examine how close $P_0$ is to the line connecting the origin and the target. For that we are going to use some vector math. The basic idea is to first compute the unit warp vector $u_w$ from the origin location $O$ to the target location $T$, then project the vector from $O$ to $P_0$ onto it, and finally use the Pythagorean theorem to find the distance from $P_0$ to its projection.

The warp vector is

$$ u_w = \frac{OT}{\lVert OT \rVert} $$

The projection of $P_0$ onto $OT$ is the point $P$ for which

$$ \lVert OP \rVert = \lVert OP_0 \rVert \cos \phi = OP_0 \cdot u_w $$

where $\cos \phi$ is the cosine of the angle between $u_w$ and $OP_0$. The distance from $P_0$ to $P$ is then

$$ d_{P0} = \lVert PP_0 \rVert = \sqrt{\lVert OP_0 \rVert^2 - \lVert OP \rVert^2} = \lVert OP_0 \rVert \sqrt{1 - \cos^2 \phi} $$

Finding the warp route with the smallest $d_{P0}$ is equivalent to finding the route that maximises $\cos \phi$. We are therefore going to work with $\cos \phi$ which can be easily computed following the definition of dot product in Cartesian space:

$$ \cos \phi = \frac{OP_0 \cdot u_w}{\lVert OP_0 \rVert} $$

Once we find the route that maximises $\cos \phi$, we will compute the distance to $P_0$. If that distance is below the range of the smartbombs, we have a winner.

from decimal import Decimal

def vec_len(v):
    return sum([x**2 for x in v]).sqrt()

def vec_norm(v):
    l = vec_len(v)
    return [x / l for x in v]

def vec_dir(p1, p2):
    return vec_norm(vector(p1, p2))

def vec_dot(v1, v2):
    return sum([x1 * x2 for (x1, x2) in zip(v1, v2)])

def vector(p1, p2):
    return [x2 - x1 for (x1, x2) in zip(p1, p2)]

def distance(p1, p2):
    return vec_len(vector(p1, p2))

"Engage!" a.k.a. Warp Physics of EVE¶

According to CCP, warp physics works quite differently from the classical Newtonian one. Most notably, the distance in metres travelled by a ship while accelerating is an exponential function of time:

$$ d(t) = \exp k t $$

where $k$ is a coefficient equal to the ship's max warp speed in AU/s. It follows that velocity (in m/s) is also an exponential function:

$$ v(t) = \dot d(t) = k \exp k t = k\,d(t) $$

The ship's speed increases exponentially once in warp until the max warp speed is reached. Deceleration follows the same exponential law with a negative and reduced in magnitude coefficient $k_d = -\min (\frac{k}{3}, 2)$, so it behaves exactly like slower acceleration in reverse time. The reduced coefficient is there so that fast warping ships do not go from "several AUs away" to "already targeting you" in the span of three seconds, giving players a chance to react meaningfully to incoming ships. All ships with max warp speed of 6 AU/s or more decelerate following different sections of the exact same speed vs. time curve with $k_d$ = -2. Ships drop out of warp when their speed reaches $v_{dw} = \min (\frac{v_{sm}}{2}, 100)$ m/s where $v_{sm}$ is the ship's max sub-warp speed, which means $v_{dw}$ is 100 m/s for the Leopard.

Since $|v(t)|$ is always $k$ times $|d(t)|$, all ships travel the same distance of 1 AU until they reach their max speed of $k$ AU/s. The same applies to the deceleration phase where all slow-warping ships (having max warp speed of no more than 6 AU/s) travel 3 AU before dropping out of warp while faster-warping ships travel $\frac{k}{2}$ AU. There are some distances covered in sub-warp acceleration and deceleration but those can be ignored given how huge of a distance AU is.

Timing¶

The time $t_a(d)$ it takes to travel distance $d$ while accelerating is found by taking the logarithm of both sides of the equation for $d(t)$ and solving for $t$:

$$ t_a(d) = \frac{\ln d}{k} $$

Nitpicking detail: This is technically the time to travel after the first metre since $d(0) = 1$ with the ship's initial speed being $v(0) = k$. Following the exponential law, it takes an infinite amount of time to cover that first metre when starting from rest. For any practical use, this is not a concern since ships enter warp with 75% of their maximum sub-warp speed which usually exceeds $k$ m/s many times. The larger initial speed results in a small correction to $d$ (i.e., $t_a$ is the travel time after some effective initial distance $d_0$), but for larger warp distances that correction is insignificant.

The time $t_a(v)$ it takes to accelerate following the exponential law from $k$ m/s (velocity at $t = 0$) to $v$ m/s is:

$$ t_a(v) = \frac{\ln \frac{v}{k}}{k} $$

Thus, the time to accelerate from initial velocity $v_0$ to $v > v_0$ is:

$$ t_a(v_0 \rightarrow v) = t_a(v) - t_a(v_0) = \frac{1}{k} \left( \ln \frac{v}{k} - \ln \frac{v_0}{k} \right) = \frac{\ln \frac{v}{v_0}}{k} $$

Technical note: As stated in the nitpicking detail above, it is mathematically impossible for warp acceleration to start from 0 m/s or deceleration to end at 0 m/s due to the infinitely long tail of the exponential function, so there must be cut-offs. $e^{kt}$ approaches 0 only asymptotically when $t \rightarrow -\infty$. This is also reflected in the equation above: the time raises to infinity as $v_0$ approaches 0.

The equation for the acceleration time applies to deceleration too, only the exponential coefficient is different:

$$ t_d(v \rightarrow v_1) = \frac{\ln \frac{v}{v_1}}{|k_d|} $$

Same applies to the remaining deceleration time as a function of the distance to the warp-out point (mius that last one metre):

$$ t_d(d_d) = \frac{\ln d_d}{|k_d|} $$

The last equation is valid when ships drop out of warp with speed $|k_d|$ m/s ($|k_d|$ times $d_d$ = 1 m). As discussed above, $v_{dw} = \min (\frac{v_{sm}}{2}, 100)$ and for most ships $|v_{dw}| > |k_d|$. Thus, there is a small correction of $-\frac{v_{dw}}{|k_d|}$ (= 50 m for the Leopard) to the deceleration distance and a corresponding correction to the deceleration time. It follows from the relationship between $d(t)$ and $v(t)$:

$$ d'_d = d_d - d_{dw} = \frac{k\,d_a}{|k_d|} - \frac{v_{dw}}{|k_d|} = \frac{k\,d_a - v_{dw}}{|k_d|} $$

and

$$ t'_d = t_d - \frac{\ln \left(\frac{v_{dw}}{|k_d|}\right)}{|k_d|} $$

The small correction to the deceleration distance can be safely discarded in most practical cases as it shifts about 50 m from the deceleration to the cruise distance, but the correction to the deceleration time is significant. For the Leopard, it means the ship drops almost 2 seconds earlier out of warp.

Let's examine the opposite case of a Leopard warping from Jita 4-4 to the Perimeter gate, which is a bit more involved. Above we showed that fast ships travel $k/2$ AU while decelerating, which is 10 AU for the zippy shuttle. Thus, $P_0$ is clearly located in the deceleration phase of the warp. To determine the full warp duration from Jita to there, we need to perform the following steps.

Step 1: Determine the full warp distance.¶

This is easy. We simply compute the distance between the warp-in and warp-out points. Here, we account for the gate's radius because the warp-out point lies on a sphere with radius equal to the target's radius. We don't know the real warp-in point because we don't know if the Leopard pilots have undock bookmarks or not. In reality, since $P_0$ is so close to the warp-out point (290654 km vs 39.8 AU), the true warp-in location has little influence on the warp time unless the journey starts from an undock bookmark that is very far away from Jita 4-4. The correction due to the radius of the target is really small but we keep it anyway since will be (ab-)using it to implement warping to a predefined distance before the target.

Step 2: Compute the time it takes to accelerate and decelerate.¶

We use the formulas for $t_a(d)$ and $t_d(d)$ but we first need to determine the acceleration and deceleration distances. For long warps, the ship reaches its max warp speed and the acceleration distance is exactly 1 AU. The decelration distance is $k/|k_d|$ times the acceleration distance (we ignore the correction due to the ship dropping out of warp with $v_{dw} > |k_d|$ because it is less than 1 km). If the warp distance is not long enough for the ship to reach its max speed, there will be no cruise phase. In that case the warp distance is:

$$ d_w = d_a + d_d = d_a + \frac{k}{|k_d|} d_a = d_a \left( 1 + \frac{k}{|k_d|} \right) $$

Thus, in general the acceleration distance is:

$$ d_a = \min \left( \frac{d_w}{1 + \frac{k}{|k_d|}}, 1 \text{AU} \right) $$

Step 3: Compute the time the ship spends cruising.¶

The cruise time is the ratio of the cruise distance and the ship's max warp speed. To obtain the distance, we simply subtract the acceleration and the deceleration distances from the total warp distance. If the ship never reaches its max warp speed, the cruise time is zero.

Step 4: Compute the time it takes to reach $P_0$.¶

We first determine in which segment of the warp the point is and then use the corresponding equation to determine the time spent in the segment, then add the time spent in the preceding segment(s).

    t_dec = d_dec.ln() / k_dec

    # Cruise distance and time
    d_cruise = d_warp - (d_acc + d_dec)
    t_cruise = d_cruise / v_cruise

    # Warp time
    t_warp = t_acc + t_cruise + t_dec
    
    if d_point <= d_acc:
        # Point is in the acceleration phase
        t_p = d_point.ln() / k_acc
        v_p = k_acc * d_point
        phase = 'acceleration'
    elif d_point <= d_acc + d_cruise:
        # Point is in the cruise phase
        t_p = t_acc + (d_point - d_acc) / v_cruise
        v_p = v_cruise
        phase = 'cruise'
    else:
        # Point is in the deceleration phase
        d_to_go = d_warp - d_point
        t_in_dec = (d_dec / d_to_go).ln() / k_dec
        t_p = t_acc + t_cruise + t_in_dec
        v_p = k_dec * d_to_go
        phase = 'deceleration'
    
    if report:
        print("Warp report:")
        print(f"  Warp distance:       {format_dist(d_warp)}")
        print(f"  Acceleration:        {format_dist(d_acc)} in {format_time(t_acc)}")
        print(f"  Cruise:              {format_dist(d_cruise)} in {format_time(t_cruise)}")
        print(f"  Deceleration:        {format_dist(d_dec)} in {format_time(t_dec)}")
        print(f"  Warp time:           {format_time(t_warp)}")
        print(f"  Distance to point:   {format_dist(d_point)}")
        print(f"  Distance left:       {format_dist(d_warp - d_point)}")
        print(f"  Warp time to point:  {format_time(t_p)}")
        print(f"  Speed at point:      {format_dist(v_p)}/s")
        print(f"  Warp phase at point: {phase}")
    
    return t_p, v_p

Getting There¶

Let's go visit the hunter's spot and say hi (or bump her, or protest somehow the killing of the magnificent bests.) Since we cannot simply warp to random 3d coordinates, we will be using bookmarks created in mid-warp.

The easiest thing to do is to board a Leopard in Jita 4-4 and set off for Perimeter, then hit the bookmark button once we are past the 6-th tick. A second later we will have the right bookmark, but also will most likely be dead and have lost 200M ISK in the process (unless we do it outside of her quite regular hunting schedule.) As there are no other ships that can do 20 AU/s, we cannot exactly reproduce the flight of the Leopard, but we can try using successive approximation. Let's write a function that shows what bookmarks we can set while warping as a function of warp distance and ship's max warp speed. We are charting the distance curve and sampling it every second along the time axis. Code is very similar to the function from the previous section, just this one uses time instead of distance and there are no comments :)

def quantize_warp_dist(warp_dist, k, v_out=Decimal(100), target_radius=Decimal(), verbose=False):
    from decimal import ROUND_UP
    
    warp_dist -= target_radius
    
    k_acc = Decimal(k)
    k_dec = min(k_acc / Decimal(3), Decimal(2))
    
    if v_out is None:
        v_out = Decimal(k_dec)

    v_cruise = k_acc * AU
    
    d_acc = min(warp_dist / (Decimal(1) + k_acc / k_dec), AU)
    d_dec = d_acc * k_acc / k_dec
    d_out = v_out / k_dec
    d_cruise = warp_dist - d_acc - d_dec + d_out
    
    t_acc = d_acc.ln() / k_acc
    t_dec = d_dec.ln() / k_dec
    t_out = d_out.ln() / k_dec
    t_cruise = d_cruise / v_cruise
    
    t_total = t_acc + t_cruise + t_dec - t_out
    
    if verbose:
        print(f"Acceleration: {format_dist(d_acc)} in {format_time(t_acc)}")
        print("+")
        print(f"Cruise:       {format_dist(d_cruise)} in {format_time(t_cruise)}")
        print("+")
        print(f"Deceleration: {format_dist(d_dec)} in {format_time(t_dec)}")
        print("=")
        print(f"Total:        {format_dist(warp_dist)} in {format_time(t_total)}")
    
    t = 0
    bookmarks = []
    
    while t <= t_total.quantize(Decimal('1.'), rounding=ROUND_UP):
        if t <= t_acc:
            d = (k_acc * Decimal(t)).exp()
            phase = 'acceleration'
        elif t <= (t_acc + t_cruise):
            d = d_acc + (t - t_acc) * v_cruise
            phase = 'cruise'
        elif t <= t_total:
            d_in_dec = d_dec - (k_dec * (t_total + t_out - t)).exp()
            d = d_acc + d_cruise + d_in_dec
            phase = 'deceleration'
        else:
            d = warp_dist
            phase = 'out of warp'
        bookmarks.append({
            'Tick': t,
            'Distance from origin': format_dist(d),
            'Distance to target': format_dist(warp_dist - d + target_radius),
            'Warp phase': phase,
            'Travel distance in m': d.quantize(Decimal('1.')),
            'Distance left in m': (warp_dist - d).quantize(Decimal('1.'))
        })
        if verbose:
            print(f"Tick {t:2d}: {format_dist(d):>15s} from start, {format_dist(warp_dist - d):>15s} till end")
        t = t + 1
    
    return pd.DataFrame(bookmarks).set_index('Tick')
